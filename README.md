# Балансировка нагрузки в веб-приложениях

Учебный проект, выполненный в рамках практического занятия №22 по дисциплине «Фронтенд и бэкенд разработка». Реализация распределения входящего трафика между несколькими экземплярами приложения с использованием Nginx и HAProxy в контейнеризированной среде.

## 🚀 Технологический стек

- **Балансировщики**: Nginx (Round Robin, Least Connections), HAProxy
- **Оркестрация**: Docker, Docker Compose
- **Backend**: Node.js, Express (кластеризованные инстансы)
- **Отказоустойчивость**: Health checks, `max_fails`, `fail_timeout`

## 📋 Выполненные задачи

### ⚖️ Практическая работа №22: Балансировка нагрузки

**Задача:** Настроить распределение запросов между серверами и обеспечить отказоустойчивость системы.

**Что было реализовано:**
- Запуск 3-х идентичных backend-серверов на разных портах в изолированной сети Docker
- Настройка **Nginx** как reverse proxy с алгоритмом Round Robin и резервным сервером (`backup`)
- Реализация проверок доступности (`health checks`) для автоматического исключения сбойных узлов
- Настройка альтернативного балансировщика **HAProxy** с базовой конфигурацией
- Оркестрация всего стека через `docker-compose.yml` (единая команда запуска)

**Особенности конфигурации:**
- `upstream` блок в Nginx для группировки бэкендов
- Параметры `max_fails=2` и `fail_timeout=30s` для быстрой реакции на ошибки
- Разделение трафика: порт 80 → Nginx, порт 8080 → HAProxy
- Использование переменных окружения для идентификации инстансов (`INSTANCE_ID`)

# API-сервер: RBAC + Redis кэширование

Учебный проект, выполненный в рамках практических занятий №11 и №21 по дисциплине «Фронтенд и бэкенд разработка». Приложение представляет собой REST API с системой разграничения прав доступа (RBAC) и кэшированием ответов через Redis для повышения производительности.

## 🚀 Технологический стек

- **Backend**: Node.js, Express
- **Аутентификация**: JWT (Access + Refresh tokens), Bcrypt
- **Кэширование**: Redis (in-memory key-value хранилище)
- **Архитектура**: Middleware-подход, модульная структура

## 📋 Выполненные задачи по практическим работам

### 🛡️ Практическая работа №11: Ролевая модель доступа (RBAC)

**Задача:** Реализовать систему авторизации с разграничением прав по ролям.

**Что было реализовано:**
- Четыре уровня доступа: `Гость`, `User`, `Seller`, `Admin`
- Middleware `authMiddleware` — проверка валидности JWT-токена
- Middleware `roleMiddleware` — ограничение доступа к эндпоинтам по ролям
- Система двойных токенов: `accessToken` (15 мин) + `refreshToken` (7 дней)
- Хеширование паролей через `bcrypt`
- CRUD-маршруты для пользователей и товаров с проверкой прав

**Маршруты с доступом:**
- `/api/auth/*` — регистрация, логин, рефреш (Гость+)
- `/api/products` — чтение (User+), создание/редактирование (Seller+), удаление (Admin)
- `/api/users` — полный доступ только для Admin

---

### ⚡ Практическая работа №21: Кэширование через Redis

**Задача:** Ускорить ответы сервера за счёт кэширования часто запрашиваемых данных.

**Что было реализовано:**
- Подключение Redis через официальный клиент `redis`
- Middleware `cacheMiddleware` — автоматическая проверка кэша перед выполнением запроса
- Функция `saveToCache` — сохранение ответов в Redis с заданным TTL
- Инвалидация кэша при изменении данных (`invalidateUsersCache`, `invalidateProductsCache`)
- Разное время жизни кэша: пользователи — 60 сек, товары — 600 сек
- Поле `source` в ответе (`"server"` или `"cache"`) для отладки

**Кэшируемые маршруты:**
- `GET /api/users` и `/api/users/:id` — TTL 1 минута
- `GET /api/products` и `/api/products/:id` — TTL 10 минут

---

```bash
# Запуск Redis (через Homebrew)
brew services start redis

# Запуск сервера
node server.js
```

# API-сервер: CRUD с PostgreSQL и MongoDB

Учебный проект, выполненный в рамках практических занятий №19–20 по дисциплине «Фронтенд и бэкенд разработка». Приложение представляет собой REST API для управления пользователями с поддержкой двух типов СУБД: реляционной (PostgreSQL) и документоориентированной (MongoDB).

## 🚀 Технологический стек

- **Backend**: Node.js, Express
- **PostgreSQL**: Sequelize ORM, pg driver
- **MongoDB**: Mongoose ODM
- **Конфигурация**: dotenv, модульная архитектура (MVC)

## 📋 Выполненные задачи по практическим работам

### 🗄️ Практическая работа №19: PostgreSQL + Sequelize

**Задача:** Реализовать CRUD API с подключением к реляционной СУБД.

**Что было реализовано:**
- Настройка подключения к PostgreSQL через Sequelize
- Модель `User` с полями: `id`, `first_name`, `last_name`, `age`, `created_at`, `updated_at`
- Unix-timestamp для полей времени (требование задания)
- Индексы для оптимизации поиска по `last_name` и `created_at`
- CRUD-эндпоинты: `POST/GET /api/users`, `GET/PATCH/DELETE /api/users/:id`
- Валидация входных данных и обработка ошибок

**Файлы:**
- `config/database.js` — инициализация Sequelize
- `models/User.js` — модель пользователя
- `controllers/userController.js` — логика CRUD-операций
- `routes/users.js` — маршрутизация

---

### 📦 Практическая работа №20: MongoDB + Mongoose

**Задача:** Реализовать аналогичный функционал для документоориентированной СУБД.

**Что было реализовано:**
- Подключение к MongoDB через Mongoose
- Схема `User` с валидацией полей и индексами
- Единая архитектура: переключение между СУБД через переменную `DB_TYPE` в `.env`
- Адаптация контроллера под особенности NoSQL (документы вместо строк)
- Поддержка `findByIdAndUpdate` с автоматическим обновлением `updated_at`

**Особенности реализации:**
- Абстракция `getModel()` в `models/User.js` для работы с обеими СУБД
- Единый формат ответа от API независимо от выбранной базы
- Минимальное дублирование кода за счёт модульной структуры

**Файлы:**
- `config/database.js` — подключение к MongoDB (Mongoose)
- `models/User.js` — схема Mongoose + индексы
- `controllers/userController.js` — универсальная логика под обе СУБД

# PWA-приложение: Заметки с напоминаниями

Учебный проект, выполненный в рамках практических занятий №13–17 по дисциплине «Фронтенд и бэкенд разработка». Приложение представляет собой прогрессивное веб-приложение для управления заметками с поддержкой офлайн-режима, push-уведомлений, напоминаний и синхронизации в реальном времени.

## 🚀 Технологический стек

- **Frontend**: Vanilla JavaScript, HTML5, CSS3
- **Backend**: Node.js, Express, Socket.io
- **PWA**: Service Worker, Web App Manifest, Push API
- **Уведомления**: Web Push, VAPID
- **Безопасность**: HTTPS (локальные SSL-сертификаты через mkcert)

---

## 📋 Выполненные задачи по практическим работам

### 🔧 Практическая работа №13: Service Worker

**Задача:** Реализовать базовый офлайн-функционал с помощью Service Worker.

**Что было реализовано:**
- Регистрация Service Worker при загрузке страницы
- Кэширование статических ресурсов (HTML, CSS, JS, иконки)
- Стратегия Cache First для статики
- Страница загружается из кэша при отсутствии интернета
- Данные заметок сохраняются в localStorage
- Добавление и просмотр заметок работают офлайн

**Файлы:**
- `sw.js` — логика кэширования и перехвата запросов
- `app.js` — регистрация Service Worker

---

**Задача:** Реализовать двустороннюю связь через WebSocket и push-уведомления.

**Что было реализовано:**
- Серверная часть на Node.js с библиотеками:
  - `express` — HTTP-сервер
  - `socket.io` — WebSocket для реального времени

- Сгенерированы VAPID-ключи через `npx web-push generate-vapid-keys`

- Клиентская часть:
  - Подключение Socket.io через CDN
  - Функции `subscribeToPush()` и `unsubscribeFromPush()`
  - Кнопки «Включить уведомления» / «Отключить уведомления»
  - Отправка события `newNote` при создании заметки
  - Получение события `noteAdded` от других клиентов
- Push-уведомления приходят даже при закрытой вкладке
- Синхронизация между вкладками в реальном времени

**Файлы:**
- `server.js` — WebSocket + Push + эндпоинты
- `app.js` — подключение Socket.io + функции подписки
- `sw.js` — обработчик события `push`

# Fullstack-приложение: Система управления товарами с RBAC

Учебный проект, выполненный в рамках практических занятий №7–12 по дисциплине «Фронтенд и бэкенд разработка». Приложение представляет собой систему с авторизацией, разграничением прав доступа и управлением каталогом товаров.

## 🚀 Технологический стек
- **Frontend**: React.js, React Router, Axios
- **Backend**: Node.js, Express, JWT, Bcrypt
- **Стилизация**: Custom CSS (Apple/Glassmorphism style)

## 📋 Выполненные задачи по модулям

### 🔐 Аутентификация и безопасность (Пр. №7–9)
- Реализована **регистрация и вход** с хешированием паролей через `bcrypt`.
- Система **двойных токенов**: 
  - `Access Token` (короткоживущий) для авторизации запросов.
  - `Refresh Token` (долгоживущий) для автоматического продления сессии.
- Защищенный маршрут `/api/auth/me` для получения данных текущего сеанса.

### 🌐 Фронтенд-интеграция (Пр. №10)
- Хранение токенов в `localStorage`.
- Настроен **Axios Interceptor**: автоматически подставляет заголовок `Authorization` и обрабатывает ошибку 401 (автопродление токена через refresh).

### 🛡️ Ролевая модель доступа (RBAC) (Пр. №11)
Реализовано три уровня доступа:
- **User**: Просмотр списка товаров и деталей.
- **Seller**: Все права пользователя + создание, редактирование и генерация тестовых товаров.
- **Admin**: Полный доступ + удаление товаров и управление списком пользователей.

## 🛠️ Основные эндпоинты API
- `POST /api/auth/register` — Регистрация
- `POST /api/auth/login` — Вход
- `GET /api/products` — Получить все товары
- `POST /api/products` — Создать товар (Seller/Admin)
- `PUT /api/products/:id` — Редактировать (Seller/Admin)
- `DELETE /api/products/:id` — Удалить (Admin)
- `GET /api/users` — Список пользователей (Admin)

## 📦 Запуск проекта
1. **Сервер**: `node app.js` (порт 8000)
2. **Клиент**: `npm start` (порт 3000)

**Тестовый аккаунт администратора:** Логин: `admin` | Пароль: `admin123`
